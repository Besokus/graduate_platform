# AGENTS.md

## 默认工作方式

- 除非用户明确要求直接修改，否则先分析再实现。
- 非平凡编码任务开始前，先识别相关模块和文件，说明当前行为、最小合理改动、风险和验证步骤。
- 优先做最小、可控、可验证的修改；不得进行无关重构或全仓库格式化。
- 遇到需求不清时先分析，不要猜测实现方向。

## Skills 与流程记录

- 开始任务前，根据用户点名或任务类型启用适用 skills，并在回复中简要说明使用的 skill 及原因。
- 多步骤开发、需求分析、复杂调试或用户明确要求记录流程时，使用项目根目录的文件记录开发过程：
  - `task_plan.md`：记录任务阶段、状态、关键决策和错误。
  - `findings.md`：记录阅读代码、测试结果、约束和发现。
  - `progress.md`：记录每次实际操作、修改文件和验证结果。
- 每完成一个阶段后，及时更新 `task_plan.md` 和 `progress.md`。
- 发现错误或验证失败时，必须记录失败命令、错误原因和后续处理方式，避免重复尝试同一失败路径。
- 最终汇报时说明使用过的关键 skills、修改范围和验证结果。

## 范围控制

- 只修改与当前任务直接相关的文件。
- 未经明确要求，不修改公共接口、数据库结构、部署配置或 CI/CD。
- 不引入新依赖，除非必要且已说明理由。
- 保持现有代码风格、包结构和项目结构。
- 不回滚或覆盖用户已有改动；如用户改动影响任务，先说明影响并协同处理。

## 多人协同开发规则

- 默认认为本项目存在多人并行开发，当前主要开发范围为题库与题目练习相关功能。
- 写代码前必须先查看当前模块及公共模块中是否已有可复用能力，包括实体、DTO、repository、service、API 封装、组件、样式和工具函数。
- 优先复用已有功能和模式；确需新增公共能力时，先说明复用不足的原因、影响范围和验证方式。
- 修改公共部分代码必须格外谨慎，公共部分包括但不限于：
  - 后端 `common` 包、认证与安全配置、全局异常、公共 DTO、用户实体与用户仓库。
  - 前端 `src/lib/api.js`、认证上下文、路由入口、全局样式、公共组件。
  - 根目录配置、构建配置、依赖配置、环境配置、数据库全局配置。
- 公共部分变更应保持最小化，不夹带题库业务实现细节，不破坏其他模块兼容性。
- 公共部分代码变更必须单独提交 Git，提交说明使用中文，并明确标注公共影响范围，例如 `fix(common): 修正认证用户来源`。
- 题库业务代码与公共代码不要混在同一个提交中；如一次任务同时涉及两类改动，先提交公共部分，再提交题库业务部分。
- 修改公共接口或共享数据结构前，必须检查调用方并说明兼容策略；无法保证兼容时先征询用户确认。
- 发现其他协作者的未提交改动或风格差异时，不得擅自回滚、覆盖或重排；只在必要范围内适配。

## 后端规则

- 遵循现有 Spring Boot 分层结构：controller、service、repository、entity、DTO。
- REST 接口统一返回 `ApiResponse`。
- 认证接口必须从 `Authentication` 获取当前用户，不信任请求体中的 `userId`。
- 业务校验放在 service 层，用户可见错误通过 `BusinessException` 返回。
- 涉及 JPA 实体字段、新表或索引时，必须在计划或结果中用中文说明数据库影响。
- 当前项目使用 `spring.jpa.hibernate.ddl-auto=update`；除非用户明确要求，不引入 Flyway 或 Liquibase。

## 前端规则

- 遵循现有 React + Vite 结构。
- API 封装集中在 `frontend/src/lib/api.js`。
- 优先复用现有布局类、组件和视觉风格。
- 中文界面文案保持 UTF-8 编码。
- 不新增前端依赖，除非用户明确批准。

## 文档书写规则

- 项目文档、开发计划、进度记录、需求说明、接口说明、测试说明和最终汇报均使用中文描述。
- 技术名词、代码标识、接口路径、命令、文件路径和 Conventional Commits 类型可以保留英文。
- 文档内容应说明“做了什么、为什么做、风险是什么、如何验证”，避免只写笼统结论。
- 新增或修改文档时，保持结构清晰，优先使用简洁标题和短列表。

## Git 提交规则

- 提交信息使用 Conventional Commits 格式，但描述必须使用中文：
  - `feat(practice): 新增练习会话与错题本`
  - `fix(auth): 修正认证用户来源`
  - `docs(agents): 更新项目协作规范`
  - `test(practice): 覆盖交卷判分逻辑`
  - `refactor(scope): 简化统计聚合逻辑`
  - `chore(scope): 调整开发配置`
- 提交正文、PR 描述、变更说明和发布说明均使用中文描述。
- 每个提交聚焦一个可验证变更。
- 公共部分代码变更必须单独提交，不能与题库业务功能提交混合。
- 提交前运行最小相关验证，并在提交说明或交付汇报中记录验证结果。
- 不提交 `dist/`、`target/`、日志、临时文件、本地配置、密钥或凭据。
- 不使用 `git reset --hard`、强推或回滚他人改动，除非用户明确要求。

## 验证规则

修改后尽量运行最小相关检查：

- 后端：`mvn test` 或聚焦测试 `mvn -Dtest=... test`。
- 前端：`npm run lint` 和 `npm run build`。
- 文档或规范类修改：至少检查目标 Markdown 文件内容和 `git diff --stat`。
- 如果无法运行验证，必须用中文说明具体阻塞原因，并给出手动验证步骤。

## 调试与性能规则

- 调试时说明症状、可能原因、优先排查路径、最可能根因和最小修复方案。
- 性能、并发、延迟或日志任务必须先识别瓶颈，区分 CPU、锁竞争、IO 阻塞、分配热点和数据结构问题。
- 并发改动必须说明线程安全影响。
- 保留必要可观测性，不做高风险重写。

## 决策优先级

当目标冲突时，按以下顺序决策：

1. 正确性
2. 稳定性
3. 可验证性
4. 最小改动范围
5. 性能
6. 可读性
